Technical Reference: Cross-Account IAM Roles and Audit Scoping

Target Audience: IT Auditors, GRC Professionals, PCI/SOC/HITRUST Assessors
Document ID: BBA-REF-AWS-01
Subject: Impact of Cross-Account Role Assumption on Identity and Access Management (IAM) Boundaries

1. Overview

In AWS, a Cross-Account IAM Role allows an entity (User, Service, or Application) in one AWS account (the Source) to perform actions in another AWS account (the Destination).

This is a standard architectural pattern for centralized management and third-party integrations. However, from an audit perspective, this configuration establishes a Trust Relationship that effectively extends the identity boundary of the Destination account into the Source account.

Official AWS Documentation: For the underlying security specifications, refer to the AWS IAM User Guide: Cross-Account Resource Access.

2. Risk Tiering: Prioritizing Audit Effort

Not all cross-account roles carry the same risk. Auditors should prioritize effort based on the permission set of the role being assumed in the destination account.

Tier 1: High-Risk (Administrative or "Admin-Like")

  • Permissions: AdministratorAccess, PowerUserAccess, or custom policies containing iam:*, organizations:*, or broad * actions on sensitive data (S3, RDS, DynamoDB).
  • Audit Requirement: Requires a full evaluation of the Source account's identity governance and authentication controls.

Tier 2: Limited / Tool-Specific Access

  • Permissions: ReadOnlyAccess, SecurityAudit, or highly scoped resource-level permissions (for example, access to a single S3 logging bucket or specific EC2 metadata).
  • Context: Common for 3rd-party SaaS tools (monitoring, cost management, security scanners).
  • Audit Requirement: Verify the role follows the "Principle of Least Privilege" and ensure the Trust Policy uses a unique ExternalId to prevent "Confused Deputy" risks.

3. The Technical Mechanism: The "Trust Handshake"

Access is governed by two independent policy decisions that must both be present for access to succeed:

  1. The Trust Policy (Destination Account): A resource-based policy attached to the IAM role that defines who is allowed to assume it (the "Lock").
  2. The Identity Policy (Source Account): A policy attached to a user or role in the source account that grants permission to call the sts:AssumeRole API (the "Key").

4. Why This Expands Audit Scope

When a Destination account trusts a Source account with Tier 1 permissions, the core identity controls typically reviewed in an audit effectively "inherit" the posture of the Source account. The auditor must verify the following in the Source environment:

  • Password & MFA Policies: Authentication happens at the point of origin. Password complexity, rotation, and MFA enforcement must be evaluated in the Source account or its connected Identity Provider (IdP/SSO).
  • Uniquely Identifiable Identities: Auditors must ensure the Source account does not use generic or shared IDs. If multiple individuals use a single Source identity to assume a Destination role, accountability and non-repudiation are lost.
  • Identity Chaining: Access is often a chain (for example, SSO User -> Source IAM Role -> Destination IAM Role). The auditor must evaluate every link, including the management of SSO permission sets and any permanent Access Keys used by automation (CI/CD).

Ownership & Responsibility Impact

  • Internal Source (Same Company): If the Source and Destination accounts are owned by the same entity, the Source account is functionally "In-Scope." A review of its IAM management, deprovisioning processes, and logging is necessary to determine the impact on the Destination environment.
  • External Source (Third-Party): If a Source account owned by a different company holds Admin-like permissions, this represents a significant security risk. This should be treated as a high-priority finding, requiring an immediate review of the third-party's access requirements and the implementation of strict "Least Privilege" controls.

5. High-Risk Configuration Patterns

Auditors should flag the following patterns for remediation:

  • The "Full Account" Trust: Principal: { "AWS": "arn:aws:iam::123456789012:root" }. This trusts the entire source account. Any administrator in the source account can grant any user the ability to assume this role.
  • CI/CD Pipeline Access: Roles assumed by automation in a separate "Dev" or "Shared Services" account that have "Write" or "Admin" access to Production.
  • Missing External IDs: Absence of an ExternalId when trusting a 3rd-party vendor, which is a requirement for many compliance frameworks to prevent cross-tenant unauthorized access.

6. Guidance for Auditors

  1. Identify Trust Origins: List all AWS accounts trusted by the environment.
  2. Filter by Risk: Separate limited 3rd-party tool roles from internal administrative roles.
  3. Audit the Point of Origin: For Tier 1 roles, request evidence of password policies, MFA enforcement, and access reviews from the Source account or IdP.
  4. Verify MFA: Ensure the Trust Policy contains a condition requiring MultiFactorAuthPresent: true for human-assumed roles.
